# MVP pivot plan — status & next steps

**Date started:** 2026-06-22
**Why:** Spent a long session iterating on algorithmic color generation (OKLCH/HSL math in `src/lib/gap-fill/material.ts`) and a rich live-preview UI. User still wasn't happy with generated palette quality after several rounds of fixes. Root cause: "does this look good" is a subjective design judgment math alone can't resolve, and we never validated whether the underlying product (tokens served via MCP) actually works with real tools. Decision: pause generation-quality polish, de-scope the UI, and validate the core product loop first.

**What stays as-is:** Database schema (`supabase/migrations/`), parsers (`src/lib/parsers/*`), MCP server (`supabase/functions/design-system-mcp/`), accessibility checker (`src/lib/a11y/`). These are considered solid — see "Already built" below.

**What's being shelved for now:** The algorithmic gap-fill color math (`src/lib/gap-fill/`) and the rich `DesignSystemPreview` UI. Not deleted — just not the focus. Revisit only after the ingestion/AI-fill loop is validated.

## The plan (5 steps)

1. **Master semantic token list → MCP → test real-tool ingestion.** Write a canonical file of all tokens with descriptive semantic names (not just primitives). Confirm the MCP serves it in a shape Figma, Tailwind, Bootstrap, and shadcn can actually ingest.
2. **Test AI ingestion of pasted variables.** Give an AI a cut-and-paste blob of a user's existing variables; see what it can map automatically and what it calls out as missing/ambiguous.
3. **AI populates missing tokens** — replaces (or augments) the deterministic gap-fill math from this session.
4. **Build UI for customizing UI elements** (after 1-3 are validated, not before).
5. **Build UI to toggle UX tokens** (date formatting, tone of voice, etc.) — start with 2-3 toggles, not the full "10 principles" list, to prove the concept cheaply. Confirm these are actually delivered correctly through the MCP.

**Decided constraint for step 3:** Don't let AI fully replace the deterministic checks. AI proposes values; the existing contrast checker (`src/lib/a11y/`) and contrast-anchored dark-CTA logic (`src/lib/gap-fill/material.ts` → `ctaDarkPair`) validate/flag anything that fails AA. Hybrid, not pure-AI — keeps this session's accessibility work as a safety net instead of discarding it.

**Monetization for now:** No paywall while testing. AI generator usage may eventually be metered (free uses included, pay for more) — not needed until the product loop is validated.

**✅ Real ingestion test — Bootstrap, done, and it found two more real bugs before passing.** Scaffolded a Node project with real `bootstrap` + `sass` (Dart Sass) installed, fetched live `framework=bootstrap` output, wrote it as SCSS `!default` overrides preceding `@import "bootstrap/scss/bootstrap"`, and ran a real compile. Two genuine bugs surfaced and were fixed in `presentBootstrap`/`toBootstrapValue` (`supabase/functions/design-system-mcp/lib/present.ts`), deployed as v7/v8:
1. **Color values were OKLCH CSS strings, which Sass can't parse.** Bootstrap's own SCSS source calls Sass-native color functions internally (`shade-color()`/`tint-color()`, used for hover states and `-subtle`/`-emphasis` variants) which call Sass's built-in `mix()` — this threw `"$color2: oklch(...) is not a color"` immediately on `@import`. Unlike Tailwind v4/shadcn (browser parses the CSS function at paint time), Sass compiles ahead-of-time and needs a value it can parse itself. Fixed: Bootstrap-specific color resolution always uses hex.
2. **Dimension values were px, causing an "incompatible units" error deep in Bootstrap's internals** — e.g. `$nav-link-height: $font-size-base * $line-height-base + $nav-link-padding-y * 2` mixes our px-based `$font-size-base` with Bootstrap's own rem-based `$nav-link-padding-y` default, and Sass refuses to add `24px + 1rem`. Bootstrap's own `_variables.scss` defaults are rem throughout for exactly this reason. Fixed: all Bootstrap dimensions convert px→rem, **except breakpoints**, which Bootstrap's own `$grid-breakpoints` map keeps in px.
3. **(Caught by a Sass warning, not yet a hard error) `$grid-breakpoints` map must be sorted ascending and start at 0** — Bootstrap asserts this at compile time. Our breakpoints were emitted in arbitrary key-insertion order with no zero floor. Fixed: `presentBootstrap` now sorts ascending and synthesizes an `xs: 0` entry if missing.

After all three fixes, the compile succeeded with **zero errors and zero warnings**. Confirmed in the actual compiled CSS output (not just absence of errors): `--bs-btn-bg: #2a6500` (our exact primary hex), `--bs-btn-hover-bg: #245600` (Bootstrap's `shade-color()` correctly darkening it — the exact function that threw the original error), `--bs-body-font-size: 1rem` (16px→1rem), `--bs-border-width: 0.125rem` (2px→0.125rem), `--bs-border-radius: 0.5rem` (8px→0.5rem). Rendered a real page (buttons, badges, alert, card) using the compiled CSS and screenshotted it — all components render correctly with our brand colors and radius.

**Status: 4 of 5 frameworks now proven through real consumers** (css-variables, tailwind-v4, shadcn, bootstrap). Only Figma remains API-shape-verified only — true verification needs the actual Figma plugin/desktop app, which isn't something drivable headlessly; would need the user's own Figma account to complete.

**✅ Found and fixed the most consequential bug yet — reported by a real external AI tool consuming the MCP, not found in-house.** User had a real vibe-coding tool (Lovable) build against `master-reference`'s `tailwind-v4` output. Lovable used `bg-primary-500` for a primary action button — a reasonable choice, since "500 is the base/main color" is the convention in Tailwind, Radix, Material, Ant Design, etc. — and got an unrelated bright lime green instead of the actual brand color (`#2a6500`, a dark forest green).

Root cause, confirmed by reading `materialColorScale` (`src/lib/gap-fill/material.ts`): the function only ever extracted **hue and saturation** from the seed color. Lightness for every step (including 500) came from a fixed, seed-independent ladder (`HSL_LIGHTNESS = [97,94,86,76,64,50,39,30,21,14,9]`) — step 500 was *always* exactly 50% HSL lightness, regardless of the seed's actual lightness. Since `#2a6500`'s real lightness is ~20%, it landed nowhere near step 500 — closer to step 800. **This isn't an edge case — it happens for almost every real brand color**, since most aren't naturally at 50% lightness. The literal brand hex a user picks essentially never appears anywhere predictable in its own generated scale.

Immediate guidance given for the live case: use `color.primary` (the flat, unsuffixed token — always the exact literal brand hex) rather than `color.primary.500` for primary actions; the numbered scale is a tint/shade ramp, not a guarantee of containing the seed.

Real fix implemented in `materialColorScale`: step 500 now always equals the seed hex exactly; the other 10 steps are proportionally interpolated FROM the seed's own lightness toward a light ceiling/dark floor (preserving the original ramp's relative spacing/shape), rather than an absolute fixed ladder. One implementation bug caught and fixed during verification: an earlier version clamped the *anchor* used for neighbor-step interpolation while keeping step 500 at the *unclamped* literal seed — for extreme seeds (near-white/near-black) this broke monotonic ordering (step 500 ended up lighter than step 400). Fixed by expanding the ceiling/floor outward to stay strictly beyond the seed's true lightness instead, so every step is computed from the exact same anchor.

Verified with 7 seed cases including the reported one and both extremes (pure white, pure black): every case hits exact step-500 match AND full monotonic ordering (checked numerically via OKLCH lightness comparison, not eyeballed). `tsc` clean, real `npm run build` passes. Regenerated `master-reference`'s primary/secondary scales (light + dark, 48 rows) with the corrected algorithm and confirmed live via the MCP: `"color-primary-500": "#2a6500"` now exactly matches `"primary": "#2a6500"` (bootstrap framework output, unambiguous hex comparison).

**This is now the most validated finding of the whole session** — found by a real third-party AI tool actually consuming the product the way a real customer would, not by us testing in isolation. Confirms the ingestion-testing strategy is working exactly as intended: validate the real-world loop, let it surface what's actually broken, then fix that instead of guessing.
